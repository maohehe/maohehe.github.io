JMM内存模型

发表于 2019-03-27 | 分类于 java ， java虚拟机 | 评论数： | 阅读次数：
本文字数： 3k | 阅读时长 ≈ 3 分钟

前导

概述

硬件上，处理器的执行速度远大于内存，大量的时间都花费在磁盘IO、数据库操作上。

如果在执行内存操作时，处理器一直等待，那将造成很大的浪费。

硬件中的并发

处理器负责计算，而对于数据的处理就要依靠和内存的交互了（读取数据，保存运算结果等）。

而两者的执行速度差距过大，如果处理器等待内存完成操作的话，会浪费很多处理器资源，所以之间加入一层高速缓存，将运算所需数据从内存复制到缓存，让处理器进行运算，运算结束再由缓存写进内存。

不过会有缓存不一致的问题。

缓存一致性问题

多处理器系统中，每个处理器都配备一个高速缓存，他们又共享同一个内存。

如果多个处理器同时运算内存中同一块区域时，就会导致各个处理器的缓存中数据不一致，那么内存中的运算处的结果数据保存哪个处理器的呢？

这就是缓存一致性问题。

Java内存模型

概述

Java虚拟机定义了一种Java内存模型，屏蔽硬件和系统的内存访问的差异，也就是Java程序在各种平台都能有相同的访问效果。

主内存与工作内存

JMM的主要目标是定义程序中各个变量的访问规则，也就是在虚拟机中将变量存储进内存和从内存中取出变量。

变量包括实例字段、静态字段、数组的元素。这些字段是在堆中的，线程共享的。变量不包括局部变量与方法参数，这些保存在虚拟机栈中，线程私有的，没有多线程竞争的问题。

JMM规定所有变量（上面说的变量）都保存在主内存中，每条线程都有自己的工作内存，工作内存保存了需要的主内存的数据的副本。线程对变量所有操作都在工作内存中进行，不能直接操作主内存的变量。

线程之间也无法互相访问对方的工作内存的变量，线程之间变量的传递都要通过主内存完成。

内存间的交互操作

一个变量从主内存拷贝到工作内存和从工作内存同步到主内存等的内存间的交互，都是规定好的，JMM中有以下 8 中操作完成。

这些操作都是原子的、不可再分的。

  1. lock：锁定。作用于主内存变量，把一个变量表示为一条线程独占状态。
  2. read：读取。作用于主内存变量，将一个变量值从主内存读取到线程工作内存中。
  3. load：载入。作用于工作内存变量，将read读取的变量载入到工作内存的变量副本中。
  4. use：使用。作用于工作内存变量，把工作内存的一个变量传递给执行引擎，供虚拟机使用。
  5. assign：赋值。作用于工作内存变量，将从执行引擎返回的值赋给工作内存的变量。
  6. store：存储。作用于工作内存变量，把工作内存变量的值传送到主内存中。
  7. write：写入。作用于主内存变量，将工作内存传来的值写入到主内存的变量中。
  8. unlock：解锁。作用于主内存变量，给锁定状态的变量解锁，解锁后才能被其他线程使用。

如果要将一个变量从主内存复制到工作内存，就要顺序执行read、load操作。

同理，把变量从工作内存同步会主内存，要顺序执行store、write操作。

注意上面的操作只要求顺序执行，并不要求紧挨着执行。例如对主内存中变量a、b进行读取载入时，read a,read b,load a,load b也是可以的，只要保证对同一个变量的read在load之前即可。

对于上面的 8 条操作也要满足一些规则，

  • read和load、store和write操作必须成对出现。
  • 只要工作内存中变量发生了assign操作，就必须同步回主内存。
  • 线程工作内存中的变量必须得被使用过，发生过assign操作，才能将值同步进主内存。
  • 新变量只能在主内存中产生，工作内存只能从主内存中复制变量值。也就是在use之前必须得先load，store之前得assign。
  • 同一时刻只有一个线程可以对变量进行lock，即lock操作是线程互斥的，但是一条线程可以重复lock，他必须提供同样数量的unlock操作。lock、unlock是成对的。
  • 不能对未被lock的变量执行unlock操作，也不能对被其他线程lock的变量执行unlock操作。
  • 对语义变量unlock之前，必须先把次变量同步到主内存才行，也即必须先执行store、write操作。

先行发生原则

上面的8中访问操作及其规则，再加上通过volatile关键字的使用，就可以确定Java程序中哪些内存访问操作是并发安全的了。但是这样的操作比较复杂。

先行发生原则可以实现同样的操作，并且简单。

概述

先行发生原则可以干什么？

可以判断数据是否存在竞争、线程是否安全。按照这个原则，通过几条规则就可以判断所有的并发环境下的两操作之间是否可能发生冲突。

先行发生是Java内存模型中定义的两个操作间的偏序关系，若操作A先行发生于操作B，A产生的影响就会被操作B观察到。

例如，

假若线程A的操作i = 1;先行发生于线程B的j = i;操作，则B操作之后，j肯定等于1，因为AB之间有先行发生规定，A对i做的操作，B可以观察到，而B发生时，C线程的i = 2; 还没有开始操作。

JMM中定义了一些先行发生的关系，天然存在的，只有这几种，

  1. 程序次序规则(Program Order Rule)：一个线程内，按照程序代码顺序，写在前面的操作先行发生于后面的操作。
  2. 管程锁定规则(Monitor Lock Rule)：一个unlock操作先行发生于后面对同一个锁的lock操作。
  3. volatile变量规则(Volatile Variable Rule)：对一个volatile变量的写操作先行发生于后面对这个变量的读操作。
  4. 线程启动规则(Thread Start Rule)：Thread对象的start()方法先行发生于此线程的其他所有动作。
  5. 线程终止规则(Thread Termination Rule)：线程的所有操作先行发生于该线程的终止检测。
  6. 线程中断规则(Thread Interruption Rule)：对线程的interrupt()方法调用先行发生于被中断的线程的代码检测到中断事件的发生。
  7. 对象终结规则(Finalizer Rule)：一个对象的初始化完成先行发生于它的finalize()方法的开始。
  8. 传递性(Transitivity)：若操作A先行发生于操作B，B先行发生于操作C，那操作A一定先行发生于操作C。

以上的 8 中规则无需借助同步手段就可以直接保证操作的顺序性。

上述操作，线程A先执行setValue方法，线程B在执行getValue方法，那么线程B获得的值是不是1?

使用先行发生规则先判断一下：

  • 两个线程的操作，规则1 不适用。
  • 没有同步操作，不需要lock\unlock，规则2不适用。
  • value没有被volatile修饰，规则3不适用。
  • 不涉及线程启动、终止、中断，也不涉及对象的终结，规则4,5,6,7不适用。
  • 没有先行发生原则，也就没有传递性，规则8不适用。

上面的8中先行发生规则都不适用，所以线程B的getValue结果是不确定的，不一定是1，也即操作是线程不安全的。
